# 2026-03-22

## 项目改造方案确定

HelloKnightRemoteCam 从"纯拍摄工具"升级为"作业管理 + 拍摄一体化平台"。

### 架构决策
- **四个进程**：Core Service（数据权威，24h常驻家庭服务器）、WeCom Agent（TS，企业微信长连接via OpenClaw）、Flutter客户端（PC按需启动）、手机服务端（不变）
- **通信方式**：Core↔Agent 用 MCP，Core↔Flutter 用 REST API + 事件推送（精确对接不需要MCP），Flutter↔手机 WebSocket（已有）
- **仓库结构**：monorepo，新增 `core/` 和 `agent/` 顶级目录
- **Core Service 技术栈**：用户倾向强类型语言，具体待定
- **开发顺序**：Phase 1 Core Service → Phase 2 Flutter任务模块 → Phase 3 MCP Server → Phase 4 WeCom Agent

### 关键设计决定
- PC端不是数据权威（不24h在线），Core Service才是
- Core Service 和 Agent 是独立进程，不合并
- Core Service 暴露两套接口：REST API（给Flutter）+ MCP Server（给AI Agent）
- 否决Web端，家长批阅通过WeCom Bot + 任务ID完成
- 企业微信用 OpenClaw 长连接模式（2026年3月企业微信新支持的特性）

## Core Service 项目骨架落地（Phase 1 第一步）

按推进板顺位 A 开始实施 Core Service 骨架。

### 技术栈选定
- **运行时**：Node.js ≥ 20
- **语言**：TypeScript（严格模式，ESM）
- **HTTP 框架**：Fastify 5（轻量高性能）
- **数据库**：better-sqlite3（同步 SQLite，适合本地服务）
- **校验**：Zod（运行时类型校验 + 自动推导 TS 类型）
- **测试**：Vitest

### 目录结构
```
core/
├── src/
│   ├── api/           # REST API 路由（task-routes + event-routes）
│   ├── events/        # 事件总线（EventBus → SSE 推送）
│   ├── models/        # 数据模型 + Zod 校验 schema
│   ├── storage/       # SQLite 持久化（database + task-repository）
│   │   └── __tests__/ # 单元测试
│   ├── config.ts      # 环境变量配置
│   ├── logger.ts      # 统一日志服务
│   └── main.ts        # 入口
├── package.json
├── tsconfig.json
└── vitest.config.ts
```

### 已实现的 API 端点
- `GET    /api/tasks` — 任务列表（支持 status/keyword 筛选 + 分页）
- `GET    /api/tasks/:id` — 任务详情
- `POST   /api/tasks` — 创建任务
- `PATCH  /api/tasks/:id` — 更新任务
- `DELETE /api/tasks/:id` — 删除任务（级联删除附件/提交物）
- `POST   /api/tasks/:id/submissions` — 添加提交物
- `DELETE /api/tasks/:id/submissions/:sid` — 删除提交物
- `GET    /api/events` — SSE 事件推送
- `GET    /api/health` — 健康检查
