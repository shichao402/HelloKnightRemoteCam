# HelloKnightRemoteCam 项目推进板

> 更新日期：2026-03-22 (Core Service 骨架落地)
> 定位：**全项目唯一推进入口**。当用户只说"继续推进 / 找下一个点 / 按项目往前做"时，先读这份文档，再决定读哪个专题文档。

## 这份文档解决什么问题

- 避免 agent 长时间卡在某个局部 checklist 里反复补细节。
- 把"现在该推进哪条主线"收敛到一个地方，而不是分散在多个专题目录里。
- 强制区分 `框架/主链推进`、`细化/优化`、`观察/维护`，优先做真正能把项目往前推一步的工作。

## 使用规则

1. 先看 `当前推进顺序`，按从上到下的顺序选条目。
2. 默认选择第一个满足下面条件的条目开工：
   - `状态` 为 `执行中` 或 `可开工`
   - 不是 `观察中`
   - 不是纯 `细化/优化`
3. 开工前先定这轮的最小目标：
   - 如果条目标明 `先规划`，先产出边界、接口、验收口径或骨架，不直接陷入细节实现。
   - 如果条目标明 `直接实现`，先落最小可运行版本，再补必要测试与文档。
4. 达到"能跑、能验、不卡下一步"就停：
   - 不继续在同一点做无明确收益的打磨。
   - 只有在阻塞下一阶段、存在真实 bug/regression、或用户明确要求时，才继续深挖同一点。
5. 收工时回写两层信息：
   - 项目级优先级、当前状态、下一步：回写本文件
   - 专题内边界、测试口径、架构细节：回写对应专题主文档
6. 收工后默认做版本留档：
   - 验证通过并完成回写后，按本轮范围做一次聚焦 `commit`
   - 若当前分支已配置可用远端，且 `push` 不会混入无关改动，则继续 `push`
   - 若存在无关脏改动、远端未配置或推送受阻，则不要硬推，必须明确说明阻塞点

## 推进原则

- 先规划，再动手。
- 先框架，再细节。
- 先跑通，再优化。
- 一轮只收口一个里程碑，不把多个主问题混做。
- 如果当前里程碑已经基本达标且不阻塞后续，就切到下一条主线，不继续局部抛光。
- 没有 failing sample、没有明确验收缺口、没有真实用户痛点时，不默认追加 heuristic 或微优化。

## 状态约定

- `执行中`：当前默认主线，优先推进
- `可开工`：没有更高优先级阻塞时，可直接接手
- `先规划`：先产出可执行方案或骨架，本轮不默认深挖实现细节
- `观察中`：只在出现真实问题、样本或触发条件时才推进
- `阻塞中`：依赖外部条件，当前不主动推进
- `已完成`：当前阶段已达到要求，默认转维护态

## 已稳定能力全景

下面按能力域归纳已交付并稳定运行的子系统。除非本看板明确重新拉起，不要再默认回去做局部补丁式打磨。

### 远程拍摄核心
- **手机服务端**：相机控制 + HTTP API + WebSocket 通知 + MJPEG 预览流 + 文件管理
- **桌面客户端**：连接管理 + 预览展示 + 拍照/录像触发 + 文件下载（断点续传）
- **共享层**：更新检查模型/服务 + 版本解析比较 + 归档/校验工具

### 工程基础设施
- **版本管理**：VERSION.yaml 单一数据源 + 脚本同步 pubspec.yaml
- **CI/CD**：build tag 触发构建 + release 脚本 + GitHub Actions + Gitee 同步
- **自动更新**：客户端拉取更新配置 + 文件名/下载地址/哈希校验

- 不默认回去优化已稳定的拍摄功能、预览功能、文件下载功能

## 回写约定

- 本文件只保留：项目级优先级、当前主线、每条主线的停止条件
- 专题文档只保留：专题内部边界、测试口径、架构决策和局部下一步
- 已完成的实现细节优先看代码、测试和 `git` 历史，不把本文件写成长日志

## 提交与推送约定

- 每轮推进达到停止条件后，默认做一次只包含本轮范围的 `commit`
- 若当前分支已配置可用远端，且推送不会混入无关改动，默认继续 `push`
- 若工作区混有无关改动、远端未配置、权限失败或 push 会带上不应提交的内容，先报告阻塞，不要为了"留档"强行提交或强推
